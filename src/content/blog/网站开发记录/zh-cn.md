# 123 天博客开发记录：一个大一新生的纯 AI 协作实验

> 从零到 932 次提交，12 个大模型协同开发，这是我的博客诞生全过程。

## 背景

我是 cmchen，一个电子信息专业的大一新生。没有任何编程基础。2026 年 2 月 16 号，我脑子一热决定用 AI 建一个属于自己的博客。123 天之后，这个博客有了 932 次 git 提交、50 多篇文章、完整的后台管理系统、旅行地图、全文搜索、中英双语……而这一切，没有一行代码是我自己写的。

全部，注意是全部，由 AI 编写。

这事说起来挺离谱的。一个学硬件的跑去搞前端，还搞得像模像样。我只能说，这年头有了 AI，。开发网站的逻辑变了。

## 我的 AI 团队

这不是一个模型单打独斗的故事。123 天里，我前后拉了 12 个模型来干活，各有各的位置：

| 角色  | 模型  | 职责  |
| --- | --- | --- |
| 架构师 | Claude Opus 4.6 | 从零搭框架，核心架构决策 |
| 主力开发 | GPT 5.3 Codex | 前期和中期所有功能开发 |
| 离谱报错修复 | GPT 5.4 Mini | 修那些莫名其妙的报错 |
| 常规报错修复 | DeepSeek V4 Flash | 修日常 bug |
| 功能开发 | DeepSeek V4 Pro | 4.28 之后的日常功能开发 |
| 重构 + 顾问 | GLM 5 | 清冗余代码 + 出主意 |
| 重构 + 顾问 | GLM 5.1 | 清冗余代码 + 出主意 |
| 前端工程师 | Gemini 3 Pro | UI 开发 |
| 前端工程师 | Gemini 3 Flash | UI 开发 |
| 前端工程师 | Gemini 3.1 Pro | UI 开发 |

分工逻辑其实很简单：Opus 定方向，GPT 5.3 Codex 写前期功能，四月底之后 DeepSeek V4 Pro 接棒日常开发，V4 Flash 修常规报错，GPT 5.4 Mini 专门对付那些离谱报错，GLM 负责清理代码顺便给建议，Gemini 三兄弟搞 UI，Kimi 当军师，GPT 5.4 负责旅行地图和 FLIP 动画。Qwen？它把我代码修坏了三次，直接拉黑。

## 第一阶段：从零到一（第 1-3 天，2026.02.16 - 02.18）

**932 次提交的前 200 多次，全部挤在这三天。** 现在回头看，那三天是我整个项目最疯狂的时候。

### Day 1（02.16）—— 诞生

Claude Opus 4.6 从零搭了整个架子。第一笔提交长这样：

* `47cb782` Initial commit
* `4f98570` 初始化 Hexo 博客，配置 GitHub Pages 自动部署
* `3b691e2` Add GitHub Actions workflow for Jekyll deployment

你没看错，一开始是 Hexo + Jekyll。但 Opus 当天就意识到不对劲，直接来了个：

* `c45f434` feat: migrate to Momo theme (Astro 框架)

这是 Opus 做的第一个关键决策：**放弃 Hexo，转向 Astro**。事后证明这个判断值千金——Astro 的内容集合、组件支持、静态生成能力，给后面所有功能铺好了路。要是死磕 Hexo，后面那些花活根本不可能。

同一天，GPT 5.3 Codex 接手，一口气肝完了：

* 双语支持（中 / 英）
* 分类系统
* 后台编辑器
* 安全跨设备认证
* 友链管理
* Header 图标（GitHub / Email）
* 邮箱复制 Toast 提示
* 博客删除功能
* 搜索快捷键和焦点陷阱
* 评论系统（加载更多、重试反馈）

第一天结束的时候，这博客已经有了基本形态：能写、能管、能搜、能聊。一个下午搞定一个博客的全部核心功能。这在 AI 出现之前根本不敢想。

### Day 2（02.17）—— 功能扩展

* 移除评论系统（加载太慢，体验不好）
* 加阅读时间显示
* 改进 PDF 转图片上传
* 分类管理优化
* 分类拆成索引页和详情页

评论系统的移除特别有意思——第一天费老大劲加上的功能，第二天就砍了。这就是 AI 开发的典型节奏：快速试错，不行就删。没有沉没成本这个概念。

### Day 3（02.18）—— 内容爆发

这一天主要是写文章和修 bug：

* 归档按月日分组
* **自动检测裸 LaTeX 公式**
* 重写自动数学公式识别器

这个自动数学公式识别器是 GPT 5.3 Codex 的杰作。它能在 Markdown 里自动找出没有被 `$` 包着的数学公式，然后正确渲染。对我来说这东西太重要了，我写高数笔记全是公式，总不能每一条都手动加美元符号吧。后来这个功能又迭代了好几次，从「检测裸公式」到「解析被包的行内 LaTeX」再到「完全重写」，每次都是因为遇到新的边界情况挂了然后又修。典型 AI 开发的节奏——能用，但总在边缘疯狂试探。

## 第七阶段：体验补完（第 140 天，07.03）

这一轮主要是补齐阅读体验和互动功能上的几个缺口。归档页、快捷键、评论、404 页四条线一起推进，全部由 GLM 5.2 完成。

### 归档页时间线增强

* `07.03` 归档页支持按年/月两级折叠，带文章计数和旋转图标
* `07.03` 文章标签按分类着色（HSL 色相槽，8 色循环），深色模式自动提亮

归档页之前是纯列表，文章一多就铺得老长。这次加了年月折叠后，整体结构清晰多了。分类着色用的是固定色相槽——同一个分类永远是同一个颜色，方便视觉定位。

### 全局快捷键

* `07.03` 新增全局键盘快捷键：j/k 上下篇导航，g h 回首页，/ 唤起搜索

快捷键的逻辑参考了 GitHub 博客。j 是下一篇（更新的），k 是上一篇（更旧的），和 vim 的上下移动一致。g 是前缀键，按 g 后 800 毫秒内按 h 就回首页。所有快捷键都会智能避让输入框和打开的模态框，不会和正常打字冲突。

## 一些反思

### 什么有效

**模型分工。** 让每个模型做它最擅长的事。Opus 做架构决策几乎从不翻车，Gemini 三兄弟做 UI 审美在线，GLM 做重构不乱动逻辑，DeepSeek V4 Pro 写功能又快又稳，V4 Flash 修常规 bug 不手软，GPT 5.4 Mini 专治各种离谱报错。各司其职比一个模型干到底效果好太多了。

**快速试错。** AI 写代码太快了，不行就删。评论系统、鼠标轨迹粒子、RSS——都是加上又删掉的。没成本，别犹豫。

**小步迭代。** 导航栏 7 次提交、移动端 8 次提交。每次只改一点点，确保不引入新问题。AI 开发最忌讳一次改太多，出 bug 了根本不知道是哪个改动引起的。

### 什么无效

**不要使用一些只看跑分的模型，Qwen 3.6 Plus。** 5 月 13 号，我让它修个小问题，结果这哥们把我整个 git 历史全毁了。DeepSeek V4 Pro 花了 40 分钟、烧了 3500 万 token 才把 git 记录捞回来，GPT 5.4 Mini 用光一周的 API 额度才把所有页面修好。从此拉黑，不再合作。

**单文件过大。** admin.astro 长到 2391 行。应该早点拆分的。


### 给其他纯 AI 开发者的建议

**选对架构师模型。** 框架搭建阶段，用你能找到的最强的模型。一个错误的架构决策，后面要花 10 倍的时间来弥补。

**报错分级处理。** 常规报错交给 DeepSeek V4 Flash 处理，那些离谱的莫名其妙报错丢给 GPT 5.4 Mini。不是所有问题都需要主力模型出来救场。

**定期做重构。** AI 写代码的习惯是不断往上堆东西。每隔一段时间让重构模型清理一次，不然代码会越来越臃肿。

**不要迷信任何一个模型。** 每个模型都有自己的盲区。交叉验证比单一模型靠谱得多。

GLM 和 Kimi 在整个过程中给了很多建议——从粒子特效到博客的定位，从功能取舍到交互逻辑。有时候我拿不准的事就去问问它们，不一定全听，但多一个视角总是好的。
